# TODO 维护流程标准（给 AI / 执行者）

最后更新：2026-03-28

这是一份“任务维护方法文档”，不是功能设计文档。  
目标是保证任何人接手一个需求时，都能按统一格式拆分、执行、验收、留痕。

## 0. 适用范围

- 适用于所有新需求、优化需求、回归修复需求。
- 适用于 AI 执行者与人工执行者协作。
- 当前唯一执行清单：`/todo/current`。

## 1. 新需求进入后的标准动作

当收到一个需求，不直接写代码，先产出一份结构化 TODO，顺序如下：

1. 目标定义（本轮要达到什么）
2. 执行规则（本轮必须遵守什么）
3. 阶段计划（Phase A/B/C... 怎么拆）
4. 验收标准（每个阶段完成的判断条件）
5. 当前优先顺序（执行顺序）
6. 基准行为项模板（用于录入行为对标项）
7. 执行日志模板（每次执行后都可追踪）

## 2. 标准 TODO 结构（强制模板）

```md

1. ...
2. ...
3. ...

## 1. 执行规则（每次都要遵守）

1. ...
2. ...

- [ ] A2 ...

1. Phase A ...
2. Phase B ...
3. ...

## 4. 基准行为项模板（后续按此录入）
```

## 3. 阶段拆分规则（怎么定义 Phase）

建议按“先标准、再对标、再差异、再建设、再验证、再发布”拆：

1. Phase A：标准与评分规则
2. Phase B：对标行为矩阵
3. Phase C：差异盘点（Gap Analysis）
4. Phase D：场景化建设（Playground/回放入口）
5. Phase E：单测基线（Vitest）
6. Phase F：E2E 回归（Vitest + Playwright）
7. Phase G：评分与门禁
8. Phase H：文档与调用链路收口
9. Phase I：优先级改造落地（P0/P1/P2）

说明：具体项目可裁剪，但必须保留“验收与回归”阶段。

## 4. 验收标准写法（统一）

每个 Phase 后必须有“验收标准”，使用可验证语句：

- 可执行：可通过 URL/命令复现。
- 可量化：如“P0 通过率 >= 100%”“e2e 通过 N/N”。
- 可追踪：能映射到测试 ID、文档章节、调用链路。

示例：

- [ ] P0 行为项全部有结论（符合/可接受/不符合）。
- [ ] 关键链路具备自动化回归（单测 + e2e）。
- [ ] `docs/usage`、`docs/developer`、`call-chains` 均更新。

## 5. 基准行为项模板（标准版）

```md
- 行为ID：
- 优先级：P0/P1/P2
- 分类：输入/选区/粘贴/评论/表格/块操作/...
- 对标来源：飞书/语雀
- 触发步骤：
- 标准预期：
- 当前行为：
- 差异结论：符合/可接受/不符合
- 风险等级：Critical/High/Medium/Low
- 修复建议：
- 测试覆盖：单测ID / e2e用例ID
- 证据：截图路径 / 日志链接
```

## 6. 当前优先顺序模板（可直接复用）

```md
## 3. 当前优先顺序（执行顺序）

1. Phase A（标准与评分规则）
2. Phase B（飞书/语雀行为矩阵）
3. Phase C（差异盘点）
4. Phase D + E + F（场景化 + 单测 + e2e）
5. Phase G + H + I（评分门禁 + 文档 + 首轮改造）
```

## 7. 执行日志模板（每次执行后更新）

```md
## 执行日志（YYYY-MM-DD HH:mm）

- 任务ID：
- 状态变更：todo -> doing / doing -> review / review -> done
- 代码改动：
- 验证命令：
- 验证结果：
- 文档更新：
- 风险与待办：
```

## 8. DoD（完成定义）

只有同时满足以下条件，任务才可标记完成：

1. 代码完成：功能可用且不破坏关键路径。
2. 测试通过：相关单测/e2e/build 通过。
3. 文档完成：`docs/usage/*` + `docs/developer/*` + `docs/developer/call-chains.md` 已更新。
4. 日志完成：`docs/todo/current.md` 已记录执行日志与剩余风险。

## 9. 给后续 AI 的执行提示（可直接复制）

```md
请按“TODO 维护流程标准”执行：

1. 先把需求转成：目标定义、执行规则、阶段计划、验收标准、执行顺序、行为项模板、执行日志模板。
2. 再按优先顺序逐阶段执行，并在每次执行后更新主 TODO 日志。
3. 每次功能完成必须同步更新 docs/usage、docs/developer、call-chains。
4. 未完成验证前不得标记 done；遇阻塞改为 blocked 并写出方案 A/B 与推荐方案。
```
